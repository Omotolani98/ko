▲ Documentation

Build distributed systems.
Not infrastructure.

Kọ́ is a Java 21+ framework for building type-safe distributed systems with declarative infrastructure annotations. Inspired by Encore.dev, built on Spring Boot 3.x. Your code is the source of truth — annotations replace YAML.

Code Generator

With Kọ́, you annotate your Java code with @KoService, @KoAPI, @KoDatabase, @KoPubSub, and other primitives. At compile time, an annotation processor builds a complete Application Model — a JSON graph of every service, endpoint, database, topic, and their relationships. The runtime engine reads this model and wires infrastructure providers per environment.

No Docker Compose. No infrastructure YAML. No boilerplate. Run ko run and your entire application starts with auto-provisioned local infrastructure via Testcontainers.

Why Kọ́?

◇

Declarative Infrastructure

Annotate what you need — databases, caches, topics — and the framework provisions it. No Terraform or Helm for local dev.

◍

Compile-Time Safety

The annotation processor validates cron expressions, API paths, naming conventions, and circular dependencies at build time, not runtime.

◎

Zero-Config Local Dev

One command spins up Postgres, Kafka, Redis — whatever your services declare. Testcontainers under the hood, no Docker Compose needed.

◆

Cloud-Agnostic

Provider interfaces abstract AWS, GCP, and self-hosted. Same code runs locally, in Docker, or across clouds without changes.

A Taste of Kọ́

UserService.java Java 
@KoService("user-service")
public class UserService {

    @KoDatabase(name = "users")
    private KoSQLDatabase db;

    @KoPubSub(topic = "user-registered")
    private KoTopic<UserRegisteredEvent> registeredTopic;

    @KoAPI(method = "POST", path = "/users")
    public UserResponse createUser(CreateUserRequest req) {
        User user = db.exec(conn ->
            conn.insertReturning(
                "INSERT INTO users (name, email) VALUES (?, ?)",
                req.name(), req.email()
            )
        );
        registeredTopic.publish(new UserRegisteredEvent(user.id()));
        return UserResponse.from(user);
    }

    @KoAPI(method = "GET", path = "/users/:id")
    public UserResponse getUser(@PathParam("id") String id) {
        return db.queryOne("SELECT * FROM users WHERE id = ?", id)
            .map(UserResponse::from)
            .orElseThrow(() -> KoError.notFound("User not found"));
    }
}
▶ Getting Started

Quickstart.

Get a Kọ́ service running in under five minutes. You'll define a service, add an API endpoint and a database, then watch the framework wire everything together.

Create a new Gradle project

Use the standard Gradle init or your IDE to scaffold a Java 21 project.

Apply the Kọ́ Gradle plugin

The plugin adds annotation processing and model generation to your build.

Define your first service

Annotate a class with @KoService and add methods with @KoAPI.

Build and inspect the model

Run ./gradlew build and find ko-app-model.json in your build output.

1. Apply the plugin

build.gradle.kts Kotlin DSL 
plugins {
    id("dev.ko")
}

ko {
    appName.set("my-app")
}

dependencies {
    implementation("dev.ko:ko-annotations:0.1.0-SNAPSHOT")
    annotationProcessor("dev.ko:ko-processor:0.1.0-SNAPSHOT")
}
Tip

The Kọ́ Gradle plugin automatically adds the annotation and processor dependencies, sets the -parameters compiler flag, and registers the model generation task.

2. Write your first service

GreetingService.java Java 
package com.example;

import dev.ko.annotations.*;

@KoService("greeting-service")
public class GreetingService {

    /** Say hello to someone. */
    @KoAPI(method = "GET", path = "/hello/:name")
    public GreetingResponse hello(@PathParam("name") String name) {
        return new GreetingResponse("Hello, " + name + "!");
    }
}

public record GreetingResponse(String message) {}

■ Core Concepts

Services.

A service is the fundamental unit of organization in Kọ́. It represents a logical microservice boundary — a group of related APIs, databases, and infrastructure that belong together.

Defining a Service

Annotate any class with @KoService to declare it as a service. The value is a kebab-case name that uniquely identifies the service across your application. This name appears in the Application Model, service discovery, tracing, and the dev dashboard.

OrderService.java Java 
@KoService("order-service")
public class OrderService {

    @KoDatabase(name = "orders")
    private KoSQLDatabase db;

    @KoPubSub(topic = "order-placed")
    private KoTopic<OrderPlacedEvent> orderTopic;

    @KoAPI(method = "POST", path = "/orders", auth = true)
    public OrderResponse placeOrder(PlaceOrderRequest req) {
        // ...
    }

    @KoCron(schedule = "0 */6 * * *", name = "expire-pending-orders")
    public void expirePendingOrders() {
        // ...
    }
}

Naming Rules

Service names must follow kebab-case convention: lowercase letters and digits separated by hyphens. The name is validated at compile time — invalid names fail the build.

NameValid?Reason
user-serviceYesKebab-case with letters
auth2YesSingle word with digit
order-service-v2YesMultiple segments
UserServiceNoUppercase not allowed
user_serviceNoUnderscores not allowed
-serviceNoCannot start with hyphen

APIs & Endpoints.

Define type-safe HTTP endpoints with @KoAPI. Request and response types are extracted from method signatures — no DTOs, no mapping, no boilerplate.

Basic API

ProductService.java Java 
/** List all products in the catalog. */
@KoAPI(method = "GET", path = "/products")
public List<ProductResponse> listProducts() {
    return db.query("SELECT * FROM products")
        .stream()
        .map(ProductResponse::from)
        .toList();
}

/** Create a new product. Requires authentication. */
@KoAPI(method = "POST", path = "/products", auth = true,
       permissions = {"PRODUCT:CREATE"})
public ProductResponse createProduct(CreateProductRequest req) {
    // Request body automatically deserialized from JSON
    // ...
}

Path Parameters

Use :paramName syntax in the path and annotate the corresponding method parameter with @PathParam. The processor validates that every path parameter has a matching method parameter.

Path parameters Java 
@KoAPI(method = "GET", path = "/products/:id")
public ProductResponse getProduct(@PathParam("id") String id) {
    // ...
}

@KoAPI(method = "PUT", path = "/products/:id/reviews/:reviewId")
public ReviewResponse updateReview(
    @PathParam("id") String productId,
    @PathParam("reviewId") String reviewId,
    UpdateReviewRequest req
) {
    // Non-path parameters are treated as the request body
}

Annotation Attributes

AttributeTypeDefaultDescription
methodString"GET"HTTP method (GET, POST, PUT, DELETE, PATCH)
pathString—URL path with optional :param segments
authbooleanfalseWhether authentication is required
permissionsString[]{}Required permissions (implies auth=true)
exposebooleantrueWhether the endpoint is public or service-internal

Request & Response Types

Kọ́ extracts type information from your method signatures. Use Java records for request and response types — their fields are automatically included in the Application Model and used to generate OpenAPI specs, TypeScript clients, and API documentation.

Request and response records Java 
public record CreateProductRequest(
    String name,
    String description,
    BigDecimal price,
    List<String> tags
) {}

public record ProductResponse(
    String id,
    String name,
    String description,
    BigDecimal price,
    Instant createdAt
) {
    public static ProductResponse from(Product p) {
        return new ProductResponse(p.id(), p.name(), p.description(), p.price(), p.createdAt());
    }
}

Design Principle

The Application Model uses snake_case for all JSON field names. This is configured via Jackson's PropertyNamingStrategies.SNAKE_CASE on the ObjectMapper, so Java records use standard camelCase field names while the JSON output is consistently snake_case.

Databases.

Declare SQL databases with @KoDatabase. The framework provisions them locally via Testcontainers and connects to managed instances in production.

Declaring a database Java 
@KoService("inventory-service")
public class InventoryService {

    @KoDatabase(name = "inventory", migrations = "migrations/inventory")
    private KoSQLDatabase db;

    @KoAPI(method = "GET", path = "/stock/:sku")
    public StockResponse getStock(@PathParam("sku") String sku) {
        return db.queryOne(
            "SELECT * FROM stock WHERE sku = ?", sku
        ).map(StockResponse::from)
         .orElseThrow(() -> KoError.notFound("SKU not found"));
    }
}

⇄ Infrastructure

Pub/Sub.

Declare message topics with @KoPubSub and subscribe with @KoSubscribe. The framework handles serialization, delivery guarantees, retries, and dead-letter queues.

Publishing

Publishing events Java 
@KoPubSub(topic = "order-placed", delivery = DeliveryGuarantee.AT_LEAST_ONCE)
private KoTopic<OrderPlacedEvent> orderTopic;

// Later in your code...
orderTopic.publish(new OrderPlacedEvent(order.id(), order.total()));

Subscribing

Subscribing to events Java 
@KoSubscribe(topic = "order-placed", name = "send-confirmation",
             maxRetries = 5, deadLetter = true)
public void onOrderPlaced(OrderPlacedEvent event) {
    emailService.sendConfirmation(event.orderId());
}

Delivery Guarantees

GuaranteeBehaviorUse Case
AT_LEAST_ONCEMessage delivered one or more times. Subscriber must be idempotent.Most use cases (default)
AT_MOST_ONCEMessage delivered zero or one times. May be lost on failure.Analytics, non-critical notifications
EXACTLY_ONCEMessage delivered exactly once (transactional where supported).Financial transactions, inventory

Caching.

Declare named caches with @KoCache. Locally backed by ConcurrentHashMap, in production by Redis or Memcached.

Using a cache Java 
@KoCache(name = "product-cache", keyType = String.class, ttl = 900)
private KoCache<String, ProductResponse> cache;

@KoAPI(method = "GET", path = "/products/:id")
public ProductResponse getProduct(@PathParam("id") String id) {
    return cache.getOrSet(id, () ->
        db.queryOne("SELECT * FROM products WHERE id = ?", id)
          .map(ProductResponse::from)
          .orElseThrow(() -> KoError.notFound("Product not found"))
    );
}
AttributeTypeDefaultDescription
nameString—Unique cache name (kebab-case)
keyTypeClass<?>String.classType of the cache key
ttllong300Time-to-live in seconds
⏰ Infrastructure

Cron Jobs.

Schedule recurring work with @KoCron. Validated at compile time.

Cron job examples Java 
// Every day at 3 AM
@KoCron(schedule = "0 3 * * *", name = "daily-cleanup")
public void dailyCleanup() { /* ... */ }

// Every 15 minutes
@KoCron(schedule = "*/15 * * * *", name = "sync-inventory")
public void syncInventory() { /* ... */ }

// Weekdays at 9 AM
@KoCron(schedule = "0 9 * * MON-FRI", name = "send-digest")
public void sendDailyDigest() { /* ... */ }

Cron Expression Format

Kọ́ uses standard 5-field cron expressions. The processor validates these at compile time — invalid expressions fail the build.

Cron format Text 
┌────────────── minute (0-59)
│ ┌──────────── hour (0-23)
│ │ ┌────────── day of month (1-31)
│ │ │ ┌──────── month (1-12 or JAN-DEC)
│ │ │ │ ┌────── day of week (0-7 or MON-SUN)
│ │ │ │ │
* * * * *
Compile-Time Validation

Invalid cron expressions like "60 * * * *" (minute > 59) or "* * *" (wrong field count) will cause a build failure with a descriptive error message.

Object Storage.

Declare storage buckets with @KoBucket. Locally backed by MinIO or the filesystem, in production by S3, GCS, or compatible stores.

Using a bucket Java 
@KoBucket(name = "user-avatars", publicRead = true)
private KoBucket avatarBucket;

@KoAPI(method = "POST", path = "/users/:id/avatar", auth = true)
public AvatarResponse uploadAvatar(
    @PathParam("id") String userId,
    UploadRequest req
) {
    String key = "avatars/" + userId + ".png";
    avatarBucket.put(key, req.data(), "image/png");
    return new AvatarResponse(avatarBucket.publicUrl(key));
}

Resolution Order

Secrets are resolved based on the environment: environment variables for local development, AWS Secrets Manager, GCP Secret Manager, or HashiCorp Vault in production — configured via infra-config.json.

⇆ Advanced

Service-to-Service Calls.

Call other services with type-safe generated clients. Same code works as in-process calls locally and HTTP calls in production.

Using a service client Java 
@KoService("notification-service")
public class NotificationService {

    @KoServiceClient
    private UserServiceClient userService;  // Auto-generated

    @KoSubscribe(topic = "order-placed", name = "notify-user")
    public void onOrderPlaced(OrderPlacedEvent event) {
        // Type-safe call to user-service — no HTTP boilerplate
        UserResponse user = userService.getUser(event.userId());
        sendEmail(user.email(), "Your order has been placed!");
    }
}
How service calls are routed
Service Client
→
KoServiceCaller
↓
Local: InProcessCaller
Direct method invocation
Prod: HttpServiceCaller
HTTP via service discovery

The KoServiceCaller interface routes calls based on the environment. Locally, when all services run in a single JVM, InProcessCaller invokes the target method directly — no network hop. In production, HttpServiceCaller resolves the target URL from service_discovery in infra-config.json and makes an HTTP call. Your code never changes.

Testing.

Kọ́ provides a JUnit 5 extension that manages infrastructure lifecycle, provisions isolated Testcontainers per test, and offers mock service clients.

InventoryServiceTest.java Java 
@ExtendWith(KoTestExtension.class)
class InventoryServiceTest {

    @KoInject
    InventoryService service;

    @KoInject
    KoSQLDatabase db;

    @Test
    void getStock_existingSku_returnsQuantity() {
        // Given: seed test data
        db.exec("INSERT INTO stock (sku, quantity) VALUES ('ABC-123', 42)");

        // When
        StockResponse result = service.getStock("ABC-123");

        // Then
        assertThat(result.quantity()).isEqualTo(42);
    }

    @Test
    void getStock_unknownSku_throwsNotFound() {
        assertThatThrownBy(() -> service.getStock("UNKNOWN"))
            .isInstanceOf(KoError.class);
    }
}
Test Naming Convention

Follow MethodUnderTest_StateOrInput_ExpectedBehavior for test method names, e.g. publish_validMessage_deliveredToSubscriber.

⚙ Advanced

Configuration.

Kọ́ uses two configuration files: ko-app-model.json (generated, read-only) and infra-config.json (environment-specific, maps logical resources to physical infrastructure).

infra-config.json

This file tells the runtime where each declared resource lives. Locally, the CLI generates it automatically from Testcontainer addresses. In production, you provide it with real connection strings.

infra-config.json (production) JSON 
{
  "metadata": {
    "app_id": "my-app",
    "env_name": "production",
    "env_type": "production",
    "cloud": "aws"
  },
  "sql_servers": [{
    "host": "db.internal:5432",
    "databases": {
      "inventory": {
        "username": "app",
        "password": { "$env": "DB_PASSWORD" }
      }
    }
  }],
  "redis": {
    "product-cache": { "host": "redis.internal:6379" }
  },
  "service_discovery": {
    "user-service": { "base_url": "http://user-svc.internal:8080" }
  }
}
Environment Variable References

Use {"$env": "VAR_NAME"} in infra-config.json to reference environment variables. The runtime resolves these at startup — secrets never appear in config files.

Provider Interface Pattern

Every infrastructure primitive follows the same pattern: a provider interface defines the contract, an in-memory implementation handles local dev and tests, production implementations handle real infrastructure (Kafka, Redis, S3, etc.), and auto-configuration selects the right one based on infra-config.json.

I

Provider Interface

Defines the contract: KoPubSubProvider, KoDatabaseProvider, KoCacheProvider, etc.

L

Local Implementation

In-memory: InMemoryPubSubProvider, ConcurrentHashMapCache, etc. Used in dev and tests.

P

Production Implementations

Real infra: KafkaPubSubProvider, RedisCache, S3BucketProvider. Separate modules.

A

Auto-Configuration

Spring Boot auto-config reads infra-config.json and wires the correct provider bean per resource.
